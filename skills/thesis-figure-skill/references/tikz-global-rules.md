# TikZ 全局约束

> **何时加载**：确定使用 TikZ 格式后，生成代码前必须加载本文件。

## TikZ 全局约束

### 语言
- 简体中文，标签简洁准确
- 不用引号，不在标签中括号补英文（除非用户要求）
- **禁止 rotate=90 中文**——xelatex 会渲染为不可读色块，所有标注水平放置

### 布局
- **字号与画布比例**：图的纵向总长度超过 12cm 时，全局默认字号必须用 `\small`（不能用 `\footnotesize`）；标注字号用 `\footnotesize`（不能用 `\scriptsize`）。纵向 ≤ 12cm 的紧凑图可以用 `\footnotesize` + `\scriptsize`。**渲染成 PNG 后检查：最小文字在 200dpi 下是否可读**
- **空白区域填充**：生成后检查是否存在大面积空白（宽度 > 3cm 且高度 > 2cm 的无内容区域）。如果存在，用注释框（note）、公式框（formula_box）、或图例（legend）填充。注释框优先放在两个主要元素之间的空白区域，而非图的最外侧
- **绝对坐标 vs 相对定位**：当需要精确控制多列等间距时（如时序图的参与方、对比图的左右栏），优先用绝对坐标 `at (x,y)` 而非 `right=Xcm of`——相对定位在多层嵌套时间距会累积偏差
- 优先垂直分层布局，控制横向宽度
- 模块间留合理间距，箭头避免交叉
- **多实例并列**用横排三列（不要纵向堆叠），列间距 2.5–3cm
- **层标题**：用独立 `\node` + `anchor` 放置（不用 `fit` 的 `label`），与相邻层框边界间距 ≥ 0.5cm；节点密集时放框内角落
- **zone 标题不能与内容框重叠**：zone 的标题（如"阶段三：结构预测"）放在 zone 的左上角或上方。但如果 zone 内有内容框（如公式框、注释框）靠近这个角落，标题就会和内容框重叠。**解决方法**：要么把标题移到 zone 的正上方（`anchor=south`），要么把内容框移开——总之标题和框之间间距 ≥ 0.3cm
- **zone 框必须足够大容纳分区标签**：对比图等含有分区标签（如"控制平面""数据平面"）的 zone 框，高度/宽度必须能放下标签文字。如果标签文字和框内节点文字重叠，说明 zone 框太小——**加大 zone 的 `inner sep` 或手动调整框的 y 范围**，不要让分区标签挤在节点文字上面
- 左右对比图中间通道 ≥ 3cm
- **标注防重叠**：同一区域内，tag 标注和 annot 标注之间至少 0.3cm；密集区域的标注优先放在节点上方或下方（而非侧面），避免与箭头路径上的其他标注冲突；如果标注过多导致无法避免重叠，删减次要标注或缩短文字
- **模块位置决定连线美观度**：如果发现某条连线需要绕很远（跨3个以上分区框），优先考虑**移动源/目标模块的位置**使其更近，而非强行画长距离绕行线。例如：IKS 模块如果要连到 M_t，应把 IKS 放在 M_t 附近（如 CPR 下方），而非放在最底层强行绕行
- **多层 zone 背景框必须左右对齐**：当图有多个分层 zone（如端-边-云-链四层架构），各 zone 因 `fit` 包裹不同节点导致宽度不同，必须用**隐形锚点**统一左右边界。做法：在所有节点之后、zone 之前，创建两个 `inner sep=0pt, minimum size=0pt` 的隐形节点 `anchor_L` 和 `anchor_R`，x 坐标分别取所有节点最左-1cm 和最右+1cm。每个 zone 的 `fit` 列表中加入 `(anchor_L |- 某层底部节点.south)(anchor_R |- 某层顶部节点.north)`，这样所有 zone 共享同一左右边界
- **层标题胶囊用绝对 x 坐标对齐**：多层分区图的左侧层标题不要用 `$(zone.west)+(-2,0)$`——各 zone 宽度不同会导致标题 x 坐标不一致。改为所有层标题共用同一个绝对 x 坐标（如 `at (-4.5, y)`），y 取该层的垂直中心
- **连线绝不穿越节点框**（反复踩坑）：画跨层连线时，必须检查路径上是否经过其他节点的矩形区域。特别是从底层上行到高层时，竖直线段容易穿过中间层的节点。解决方法：计算路径上所有 y 值对应的 x 范围内有哪些节点，如果有冲突则将出发点的 x 坐标移到两个节点之间的空隙中
- **短距离弯折连线走空白侧**：当源节点和目标节点在垂直或对角方向相邻（间距 < 3cm），连线不要从源节点正下方/正上方直接弯折到目标——这样弯折点容易紧贴节点边缘或和其他连线挤在一起。正确做法：从源节点出发后**先水平走到附近的空白区域**（远离其他节点和连线的一侧），再转弯到达目标。用 `|- ` 或 `-|` 保证最后一段完全水平/垂直进入目标锚点，避免微小斜线

### 代码规范
- `standalone` 文档类，`border=25pt`（有外部回路时 35pt）
- **CJK 字体可移植性**（关键教训）：
  - 编译前**必须** `fc-list | grep "字体名"` 确认字体存在
  - 多平台字体对照表：

    | 平台 | 首选字体 | 备选字体 | LaTeX 配置 |
    |------|---------|---------|-----------|
    | macOS | PingFang SC | Heiti SC | `\setCJKmainfont{PingFang SC}` |
    | Linux | Noto Sans CJK SC | WenQuanYi Micro Hei | `\setCJKmainfont{Noto Sans CJK SC}` |
    | Windows | Microsoft YaHei | SimHei | `\setCJKmainfont{Microsoft YaHei}` |

  - ⚠️ xelatex 对缺失字体**静默失败**：仍生成 PDF 但中文全丢（只有 Missing character warning，无 error）。编译后必须 `grep "Missing character" *.log` 检查
  - 中文支持优先 `ctex`（`\usepackage[fontset=none]{ctex}` + 上表字体），不可用则退回方案 B：`\usepackage{fontspec}` + `\setmainfont{...}` + `\setsansfont{...}`
  - **不要用 `ucharclasses` 方案**——实测中 `\setDefaultTransitions` 在 tikz 节点内的中英混排场景下频繁出现 Missing character 错误
- 所有节点用 `style` 定义 + `positioning` 相对定位（**几何图例外：允许绝对坐标**）
- 编译：`xelatex -interaction=nonstopmode`
- 箭头旁标注用 `tag` 样式（无背景或半透明背景），独立标注用 `annot` 样式（不带 `fill`）
- **箭头头大小必须匹配箭身长度**——基线参数见 `lessons.md` Part 1「箭头」表。短箭头场合必须用内联 scale 覆盖全局 style，不要让长箭头的 scale 跑去画 0.5cm 短箭头。
- **箭头终点必须精确止于目标框边界**：硬编码箭头终点时，y/x 值必须等于目标框 `rectangle` 的边界坐标。框顶 y=3.0 则箭头终点 y=3.0，不能 y=2.6（刺入框内）也不能 y=3.3（悬空）。审查时逐条指向框的箭头对照框坐标。
- **嵌入图表坐标轴标签防重叠**——参数见 `data-visualization.md` 「坐标轴标签防重叠」表。生成嵌入图表时必须主动给轴标题加 shift，不要等渲染后才发现重叠。
- **节点补充说明必须放在框内**（用 `\\` 换行 + `\scriptsize` 第二行），不要放在框正下方——框下方的标注会被向下的连线穿过，这是最常见且最顽固的排版问题。如果框太小放不下，增大框的 `minimum height`，不要把标注外挂到框外。**审查时必须逐个框检查**：这个框下面/上面/左右有没有独立的 `\node` 标注？如果有，它会不会被连线穿过？只要有这种风险就立刻合并到框内
- **反馈虚线与 zone 边框虚线不能在同一区域交叉**——两者都是 dashed 线，颜色可能相近，在交叉区域读者分不清哪条是反馈线哪条是边框。解决方法：反馈线的 rail x 坐标必须在 zone 边框**外侧至少 1.0cm**，确保反馈线和 zone 边框在空间上完全分离，不在同一个视觉区域内
- **普通跨层连线必须用单条 `\draw` + 路径点画连续弧线**，禁止拆成多段 `\draw`——多段拆分会在弯折点产生箭头头/尾拼接的丑陋痕迹。正确做法：
  ```latex
  % ✅ 正确：单条 \draw，弯折自然连续
  \draw[arrow, rounded corners=6pt]
      (source.south) -- (source.south |- 0,-5) -- (target.west |- 0,-5) -- (target.west);
  % ❌ 错误：拆成多段，弯折点有箭头拼接
  \draw[arrow] (source.south) -- (0,-5);
  \draw[arrow] (0,-5) -- (target.west);
  ```
- **只有长距离回路（U型绕行 > 10cm）才拆分为 3 段独立 `\draw`**——因为单条 `\draw` + `rounded corners` 画长距离回路时路径可能异常。拆分时中间段不带箭头（只有最后一段带箭头）：
  ```latex
  \draw[dashed, thick, color=drawRedLine, rounded corners=6pt]
      (source.west) -- (-3, source_y);  % 段1：无箭头
  \draw[dashed, thick, color=drawRedLine]
      (-3, source_y) -- (-3, target_y);  % 段2：竖直段，无箭头
  \draw[-{Stealth}, dashed, thick, color=drawRedLine, rounded corners=6pt]
      (-3, target_y) -- (target.west);  % 段3：带箭头
  ```
- 回路终止于分区框边界（如 `target_zone.west`），不穿入框内
- 多条回路分配到不同侧，同侧间距 ≥ 1.5cm
- 跨区箭头在空白区域中转（L 型路径），不直接刺穿虚线框
- **层标题与箭头标注防冲突**：当垂直箭头穿过层间空隙时，箭头 tag 标注和层标题必须放在不同侧——标注在 `right` 则标题用 `anchor=north west`（左下角），反之亦然。生成后必须检查每个层标题附近是否有箭头标注，有则错开
- **同侧多条回路/跳连的 rail 分配**：最外侧 rail 距框边界 ≥ 2.0cm，内侧 rail ≥ 0.8cm，两条 rail 间距 ≥ 1.0cm。每条回路用不同颜色区分。禁止两条回路共用同一 rail x 坐标
- **跨层连线不要默认走右侧 rail——先看有没有更短的路径**：AI 有个坏习惯：所有跨层反馈/下发线都走右侧 rail 绕行。但如果源和目标在同一 x 列或相邻列，**一条直连或简单 L 型就够了**——不需要绕到画面最右边再绕回来。**只有当直连/L型会穿越其他节点时**才走外侧 rail。例如"聚合层→边缘层"如果两者的某个模块在同一 x 上，直接垂直虚线下来就行，不要绕右侧。**两条虚线绝不共享同一段 rail**——如果两条线必须走同侧，用不同 x 坐标的 rail（间距 ≥ 1.5cm）
- **连线路径需要空间推理，不是死规则**：画每条连线前，在脑中模拟 2-3 种可能的路径，评估每条路径会穿过哪些元素、在哪里弯折、标签放在哪里最自然，然后选最优的那条。具体方法：
  1. 看图中的实际布局，找到源和目标之间的所有**空白通道**（框与框之间的间隙、zone 外侧、底部通道）
  2. 在脑中沿每条候选路径"走一遍"——经过哪些区域？会不会穿过节点文字？弯折点附近有没有拥挤？
  3. 选择穿越元素最少、弯折最自然、标签最好放的那条。**路径越短越好**——如果两个节点在同一 x 坐标上，直接 `.south` → `.north` 直连，不要绕弯；如果斜向相邻，一个简单的 L 型弯折就够了，不要绕大 U 型。**直连永远优先于绕行**
  4. **进入目标节点的方向也要选最优的**——不是固定从 `.south` 或 `.east`，而是看哪个方向进入时不会穿过其他元素
  5. **出发锚点也要选最优的**——不要所有线都从 `.south` 出发导致重叠。如果两条线从同一节点出发去不同方向，用不同锚点（如 `.south west` 和 `.south east`，或 `.south` 和 `.east`）。**看目标在哪个方向，就从那个方向的锚点出发**——目标在右下方就从 `.south east` 或 `.east` 出发，不要从 `.south` 出发再绕弯
- **虚线/辅助线（如损失函数连线、对比学习连线）绝对不穿越分区框**：这是最常见的遗漏。当虚线需要从一个分区内的节点连到分区外的标注时，必须先从节点侧面出框（`.west` / `.east` / `.south`），在分区框边界外侧空白区域中转，再到达目标。绕行路径用 `rounded corners=6pt` 保持美观。宁可路径长一点也不穿框
- **绕行 rail 必须与分区框边界保持 ≥ 1.0cm 间距**：如果 rail 紧贴框边界（如只有 0.5cm），虚线和框的虚线在视觉上会混淆不清。**跨越多个分区的长距离 rail 必须放在画面中所有元素（包括独立节点如 Training Data）的最外侧**，而不仅是某个分区框外侧。计算 rail x 坐标时，取所有相关节点和分区框的最左/最右边界，再外推至少 1.0cm
- **虚线出口和入口方向选择**（反复踩坑的规则）：出口方向必须指向空白区域，不能指向其他节点所在方向。例如：从 VAE 连到右下方的损失标签，应从 VAE `.south` 向下出框底部再水平右行，而非从 `.west` 水平左行（会穿过左侧节点文字）。入口方向优先选择目标节点的**空闲侧**（没有其他连线的一侧），避免进入方向与已有箭头重叠
- **虚线出口必须留足间距**：从节点出来后至少走 **1.0cm 以上的直线段**再转弯，否则转弯点紧贴节点边缘看起来很突兀。入口处同理——到达目标节点附近时，先在旁边转弯对齐，再直线进入，形成清晰的 L 型弯折
- **连线出口/入口选择节点的自然锚点**：向下连接用 `.south`（底部中间），向右用 `.east`，向左用 `.west`。禁止从角落（`.south west` / `.north east`）出发，角落出口看起来不自然。入口同理——从目标节点最近的正面（`.west` / `.north` / `.east`）进入，不从角落进

### 箭头方向感（最常被 AI 忽略的问题）

**核心原则**：每条箭头的**视觉流向**必须和**逻辑流向**一致。人看到一条箭头时，会在 0.1 秒内直觉判断"这条线在往哪走"——如果这个直觉方向和箭头实际表达的数据流方向矛盾，读者会觉得"这个箭头指反了"，即使箭头最终连到了正确的目标。

**产生"看起来反了"的根本原因**：出发锚点或到达锚点选择错误。

**正确的锚点选择原则**——出发方向 = 目标所在方向：
- 目标在源的**正上方** → 从 `.north` 出发
- 目标在源的**正下方** → 从 `.south` 出发
- 目标在源的**右边** → 从 `.east` 出发
- 目标在源的**左上方** → 从 `.north` 出发，用 L 型弯折水平段左行到达目标
- 到达锚点同理：箭头应从**源所在方向**进入目标（源在下方则从 `.south` 进入，源在左侧则从 `.west` 进入）

**典型错误案例**：
```
❌ 相机在点云左边，但连线从相机 .east 出发绕到点云 .east 进入
   → 视觉上像是"先往右走再折回来"，方向混乱
✅ 相机 .east → 点云 .west（直接向右流）
   或 相机 .north → 上行 → 点云 .north（向上流再向右）

❌ 多条线从同一节点出发，为避免重叠把某些线换到了反方向的锚点
   → 虽然避免了重叠但方向感全乱了
✅ 用同方向的相邻锚点（如 .north 改为 [xshift=-5pt]node.north 和 [xshift=5pt]node.north）
   或微调目标锚点的 yshift/xshift 错开入口
```

**自检方法**：代码写完后，逐条 `\draw` 检查——出发锚点的方向是否朝着目标？到达锚点的方向是否朝着源？如果一条从下层到上层的数据流线从 `.east` 出发，那它的第一段线段就是水平的，读者会觉得"怎么先往右走了？"——应该从 `.north` 出发让第一段是向上的。

**最阴险的方向 bug：中间点超越目标**：
`\draw[arrow] (A) -- (mid) -- (B)` 中，如果 mid 的 y 超过了 B 的 y（如 A 在 y=5.1，mid 在 y=5.65，B 在 y=5.4），最后一段从 5.65→5.4 方向反转，箭头尖朝下——但逻辑上数据应该向上流。**这不是锚点选错，而是路径规划 bug**。修复：同轴路径删掉多余中间点直连 `(A) -- (B)`；或确保 mid 的值严格在 A 和 B 之间。

```
❌ \draw[arrow] (pe.north) -- (-5.5, 5.65) -- (mha.south);
   % pe.north=5.1, mid=5.65, mha.south=5.4 → 5.65>5.4 → 箭头朝下！
✅ \draw[arrow] (pe.north) -- (mha.south);
   % 直连，5.1→5.4 单调递增 → 箭头朝上 ✓
```

### 一对多分叉连线（树状扇出）

当一个源节点连接到**多个水平排列的目标节点**时（如骨干网络→多个特征尺度），**禁止从同一个锚点画多条斜线扇出**——斜线不平行、不垂直，很难看。

**正确做法：树状分叉**（弯折处加 `rounded corners`）：

**茎干必须从源框正中心垂直出发**。以下模板**必须逐行照抄**，只替换节点名和 `横线y` 值，不可修改结构：

```latex
% ===== 树状扇出完整模板（2个目标）=====
% 步骤0：计算横线y（源框底部往下 0.8cm，且距目标 >= 1.5cm）
% 假设 source.south 的 y ≈ Y_src，target.north 的 y ≈ Y_tgt
% 横线y = Y_src - 0.8（茎干长度）
% 检查：Y_tgt - 横线y >= 1.5cm，否则加大源和目标的间距

% 步骤1：茎干——从源中心垂直下降到横线高度
% ⚠️ 必须用 source.south 而不是手写坐标
\draw[thick, color=black!70]
    (source.south) -- (source.south |- 0, 横线y);

% 步骤2：左分支——从茎干底部出发，用 let 提取目标 x 坐标确保垂直
\draw[arrow, rounded corners=5pt]
    let \p1=(target_left.north) in
    (source.south |- 0, 横线y) -- (\x1, 横线y) -- (target_left.north);

% 步骤3：右分支——同理用 let 提取
\draw[arrow, rounded corners=5pt]
    let \p1=(target_right.north) in
    (source.south |- 0, 横线y) -- (\x1, 横线y) -- (target_right.north);
% ===== 模板结束 =====
```

**关键点**：所有分支的横线起点都是 `(source.south |- 0, 横线y)`——这个点的 x 坐标由 `source.south` 决定（正中心），y 坐标是横线高度。**禁止**用 `(source_x, 横线y)` 手写 x 值——手写 x 值是树歪掉的头号元凶。

**源节点必须水平居中于目标节点组**：树状扇出要好看，源节点的 x 必须是目标节点组的水平中点（`源.x = (最左目标.x + 最右目标.x) / 2`）。如果源偏了，左右分支长度不同，树就歪了。**布局规划阶段就要确保对齐**——把源节点放在目标行的正上方中心。如果源无法居中（被其他 zone 约束），不要用树状扇出，改用独立 L 型。

**双向通信（上行+下行共享同一组节点）用双偏移树**：当 A→B 和 B→A 两个方向都需要一对多连线时（如联邦学习的梯度上传+模型下发），画两棵树但**茎干微偏、横线错高、分支接不同侧**：
- **树A（上行汇聚）**：茎干在 `[xshift=-0.3cm]source.south`，横线在 y1（靠近目标），分支接目标的 `[xshift=-8pt].north`（左侧）
- **树B（下行分发）**：茎干在 `[xshift=+0.3cm]source.south`，横线在 y2（靠近源，y2 ≠ y1，间距 ≥ 1.0cm），分支接目标的 `[xshift=+8pt].north`（右侧）
- 两棵树形状都完整对称，茎干偏移仅 0.3cm（肉眼几乎看不出不对称），横线和分支完全不重叠
- **标签放在图两侧空旷区域**，不能放在两棵树之间——否则一定会和某条线重叠
- **禁止**用 `xshift=±5pt`（太小，线重叠）或 `xshift=±1cm`（太大，树歪得明显）。0.3cm 是经验证的最佳平衡值

**树状扇出只用于向下方向**（关键审美规则）：
- **向下扇出**（父节点在上，子节点在下）：自然、好看——符合"树"的生长方向和重力隐喻。箭头从横线向下伸到子节点，视觉上像树枝分叉
- **向上扇出**（源在下，目标在上）：**禁止使用**——倒立的树极其不自然。横线上方伸出的弯折箭头看起来"拐过来就直接戳进框底"，很丑
- **向上一对多的正确做法**：不用树状扇出，改用**独立直连**。从源节点 `.north` 出发一条垂直茎干到中间高度，然后每条分支独立走 L 型路径到各目标的 `.south`。或者如果源和某个目标 x 坐标对齐，直接垂直直连那个目标，其余目标走独立 L 型
- **多对一向上汇聚**（多个源在下，一个目标在上）：用**反向树**是可以的——多条线从各源 `.north` 向上汇聚到一条横线，再从横线垂直向上到目标。这个方向看起来像"多条河流汇入一个点"，视觉上自然

**横线到目标框的距离 ≥ 1.5cm**（仅适用于向下扇出和向上汇聚）：
- `横线y = 源.south的y - 0.8cm`（茎干），且横线到目标 `.north` 的距离 ≥ 1.5cm。如果目标框距源太近（< 2.3cm），增加间距或改用直连
- **为什么 1.5cm**：`rounded corners=5pt` 弧线过渡"吃掉"约 0.5cm，1.5cm 确保弧线后仍有 ≥ 1.0cm 可见直线段

**垂直段必须严格垂直**——树状扇出中最常见的问题是垂直段有微弯。原因：分支的起点 x 坐标（横线上的点）和终点 x 坐标（目标 `.north` 的 x）不一致。**必须用目标节点的 x 坐标作为分支起点的 x**。用 TikZ 坐标提取确保精确：`let \p1=(target1.north) in (\x1, 横线y) -- (target1.north)`。

**同理适用于多对一汇聚**：多个源汇聚到一个目标时，用反向树状结构（三条垂直上升→横线→一条垂直下降到目标）。汇聚横线到源 `.north` 的距离同样 ≥ 1.5cm。

**汇聚树的箭头规则——分支不带箭头，只有树干带**：汇聚树中，从各源到横线的分支线用 `\draw[thick]`（**无箭头**），只有从横线到目标的树干用 `\draw[arrow]`（**带箭头**）。如果分支也带箭头，多个箭头尖在横线上"对撞"（`>-< ↓`），极其难看。正确的视觉语义是"多条河流汇入一条干流"——河流本身不需要箭头，只有干流入海处才有方向。扇出树同理：树干不带箭头，分支带箭头指向各目标。

**树干 x = 中间源的 x，中间源的线一路笔直到底**：汇聚树最自然的画法是让中间源（居中的那个）的垂直线直接延伸为树干——从源框底部一路笔直向下穿过汇聚横线到目标，没有任何水平偏移。左右两侧的分支用圆角弯折汇入这条中线。这样中间那条线是完全笔直的，视觉上最稳定。如果目标框中心和中间源的 x 略有偏差（如 0.2cm），树干就进入目标框稍偏的位置——目标框足够宽，0.2cm 偏移看不出来，远好过让中间线歪掉。

**多箭头入同一节点——分散锚点，不要挤在一起**：当 3 条以上连线需要进入同一个节点时，不要全用 `yshift` 微调同一个锚点（如 `.west` + yshift=±3pt/±6pt/±9pt）——挤在一起会让 `rounded corners` 产生非90度的扭曲变形。**正确做法**：从不同方向进入，用不同锚点（`.north`、`.west`、`.north west`、`.south west` 等）。每条线有独立的弯折路径，弯折之间互不干扰。如果确实需要同侧进入，锚点间距 ≥ 0.5cm，每条线的弯折走廊间距 ≥ 0.8cm。

### 多源汇聚→多目标扇出（沙漏树）

当多个源节点需要连接到多个共享目标节点时（如对比图的两个结果→共享评估指标），**用树状汇聚-扇出代替点状 bus**。点状 bus（水平线+圆点连接点）显得机械别扭，树状连接更流畅自然。

**结构**：汇聚树（上半）+ 树干 + 扇出树（下半），整体形成沙漏形：

```latex
% === 汇聚树：多个源 → 合流到中心（横线在 y=-0.8）===
\draw[thick, color=black!50, rounded corners=5pt]
    let \p1=(sourceA.south) in
    (sourceA.south) -- (\x1, -0.8) -- (0, -0.8);

\draw[thick, color=black!50, rounded corners=5pt]
    let \p1=(sourceB.south) in
    (sourceB.south) -- (\x1, -0.8) -- (0, -0.8);

% === 树干：合流点 → 扇出点 ===
\draw[thick, color=black!50]
    (0, -0.8) -- (0, -2.0);

% === 扇出树：中心 → 多个目标（横线在 y=-2.0）===
\draw[arrow, color=black!50, rounded corners=5pt]
    let \p1=(target1.north) in
    (0, -2.0) -- (\x1, -2.0) -- (target1.north);
% ... 对每个目标重复

% === 标题覆盖树干（画在连线之后，z-order 最高）===
\node[font=\small\bfseries, fill=white, inner sep=5pt]
    at (0, -1.4) {Evaluation Metrics};
```

**关键要点**：
- 汇聚横线的中心 x = 所有源节点的 x 中点（树干垂直、左右对称）
- 如果树干中间有标题文字，标题用 `fill=white` 覆盖树干线，且必须画在所有 `\draw` 之后
- 扇出横线到目标 `.north` 的距离 ≥ 1.5cm（同普通树状扇出规则）
- 汇聚部分不带箭头（只是汇流），扇出部分带箭头（指向目标）

### "能直就直"原则（最基本但最常违反的原则）

**如果源和目标在同一 x 或同一 y 上，直接用一条直线连接。不要画 L 型弯折。**

AI 画连线时有一个坏习惯：无论什么情况都用 L 型路径（先水平再垂直，或先垂直再水平）。但当两个节点**完全对齐**时，L 型路径会产生一个完全多余的弯折——读者会困惑"为什么这条线要绕一下？"

**判断方法**：
1. 源和目标的 x 坐标相同 → 纯垂直直线 `(source.south) -- (target.north)`
2. 源和目标的 y 坐标相同 → 纯水平直线 `(source.east) -- (target.west)`
3. 只有 x 和 y **都不同**时才需要 L 型弯折

**弯折后的线段长度**：弯折点到目标框之间的距离 ≥ **1.5cm**（而非 0.8cm）。原因：`rounded corners=6pt` 的弧线过渡本身会"吃掉"约 0.5-0.7cm 的线段，如果弯折点距目标只有 0.8cm，圆角过渡后实际可见直线段只剩 0.1-0.3cm，箭头看起来"刚拐过来就到了"，非常丑。1.5cm 的距离确保圆角过渡后仍有 ≥ 0.8cm 的可见直线段。**渲染后放大检查每个弯折→箭头的过渡**：箭头前是否有一段足够长的直线段？

**draw.io 中同样适用**：draw.io 的 orthogonalEdgeStyle 有时会给对齐的节点添加不必要的弯折。手动设置 waypoint 确保对齐节点之间是直线。

**审查方法**：渲染 PNG 后，沿每条箭头从起点到终点用眼睛"走一遍"。第一段线段的方向就是读者对这条箭头的第一印象——它必须和逻辑方向一致。

### 中心辐射图（Hub-Spoke）布局规则

中心辐射图用于展示一个核心模块与多个周围模块的交互关系。

- **环形分布必须均匀**：N 个 spoke 节点用极坐标均匀分布在同一圆环上，角度间隔 = 360°/N。用 `\foreach \i/\name/... in {0/A, 60/B, 120/C, ...}` 循环生成。**禁止**手动摆放——容易变成十字形或不均匀分布
- **圆环半径**：4-5cm，使 spoke 节点之间不会太挤也不会太远。中心节点放在 (0,0)
- **中心节点突出**：尺寸比 spoke 大 1.5 倍以上，用绿色或其他醒目色，加粗边框（`line width=1.5pt`）
- **连线方向**：从 spoke 节点指向中心用 `.center` 锚点方向（不是固定用 `.south` 或 `.west`），用 `(spoke) -- (center)` 自动选最短路径
- **spoke 间连线必须克制**：hub-spoke 图的价值在于清晰的辐射结构。spoke 间辅助连线（闭环、支持线等）**最多 2-3 条**——超过就变成蜘蛛网，读者根本看不出核心结构。如果确实有很多 spoke 间关系，只画最核心的闭环路径（如 A→B→C→A），其余用注释框文字补充。**辅助连线（dotted/dashed）不能穿越中心区域**——走环外侧弧线，否则和 hub-spoke 主线交叉成一团
- **异步/事件连线**：spoke 之间的直接通信用弧线（`bend left=20` 或 `bend right=20`）走环外侧，**多条弧线不能交叉**——交叉时调整弯曲方向或角度使其分离
- **视觉区分**：如果 spoke 节点有不同类别（如读写型 vs 通知型），用不同颜色区分，不要全部同色

### 多实例汇聚图布局规则

多实例图（如联邦学习、分布式训练）展示多个相似节点与中央节点的交互。

- **PS 与 Worker 间距紧凑**：PS 到 Worker 顶部的垂直距离控制在 3-4cm，不要超过 5cm。中间区域只有连线，不需要太多空间
- **上行线和下行线路径分离**：上传梯度（实线）和广播模型（虚线）走不同路径——左右两侧 Worker 用不同 y 高度的水平走廊，中间 Worker 可以直连。两类线的走廊间距 ≥ 1cm 确保清晰区分
- **对称线必须严格平行**：同一对节点之间的上行和下行线应该**严格平行**（垂直段 x 坐标相同或等间距偏移）。如果迭代修改后平行关系被破坏，必须恢复——**迭代过程中不能让美观度退步**，修复一个问题不能引入另一个审美问题

### 连线弯折质量（关键教训）

**所有连线必须是"纯正交 L 型"**——即每段线段要么纯水平要么纯垂直，禁止斜线段。**弯折处使用 `rounded corners=6pt`**（或 8pt），避免死板的直角——圆角弯折更自然美观。

**常见弯折问题及修复**：

| 问题 | 表现 | 原因 | 修复 |
|------|------|------|------|
| 斜线穿越 | 连线从 A 斜穿到 B | 直接 `(A) -- (B)` 但 A/B 不在同一行/列 | 改为 `(A.south) |- (B.west)` 或 `(A.east) -| (B.north)` |
| 尖锐直角 | 弯折点紧贴节点边缘，或直角过于生硬 | 弯折距离太短（< 0.5cm），或缺少 rounded corners | 先走 1.0cm 以上直线段再弯折，且加 `rounded corners=6pt` |
| 多段锐角折线 | 连线有3个以上弯折 | 路径规划不当 | 重新设计路径，最多2个弯折（L型=1个弯折，U型=2个弯折） |
| `-|` 退化 | 源和目标在同一 x 上时 `-|` 产生多余水平段 | TikZ 的 `-|` 语义不适用 | 改为纯垂直 `--` 直连 |
| 弯折穿越节点 | 弯折后的水平/垂直段穿过其他节点 | 弯折点的 x 或 y 在其他节点范围内 | 将弯折点移到两个节点之间的空隙 |
| 隐式斜线 | 看似直线但实际微斜 | `++(dx,dy)` 与绝对坐标 `(x,y)` 在同一 `\draw` 中混用 | **跨层 rail 连线全部使用绝对坐标，禁止 `++` 与绝对坐标混用** |
| 非90度弯折（**最顽固的问题**） | 弯折处不是标准90度，出现奇怪角度 | 路径中间点的 x 或 y 和目标锚点没精确对齐 | **路径中每个中间点必须与前一个点共享 x 或 y**。例如从 `(2, 10)` 到 `(7, 8)`，中间点必须是 `(2, 8)` 或 `(7, 10)`——不能是 `(3.5, 9)` 这种两个坐标都不同的点。**代码提交前强制验证**：列出每条 `\draw` 路径的所有坐标点序列，逐对检查是否共享 x 或 y。**特别注意**：当使用 `node.south` 等锚点和手写坐标 `(x, y)` 混用时，锚点的实际坐标可能和你预期不同——用 `let \p1=(node.south) in` 提取实际坐标来构造中间点 |

**渲染后检查连线弯折**：在 PNG 中逐条检查：
- [ ] 每条连线是否全部由水平段和垂直段组成（无斜线）
- [ ] 弯折点是否远离节点边缘（距离 ≥ 0.5cm）
- [ ] 弯折次数是否 ≤ 2（L型1次，U型2次）
- [ ] 弯折后的线段是否穿过其他节点
- [ ] **箭头碰撞检查**：两条连线是否在同一区域交叉、平行太近（< 0.3cm）、或共享同一路径段？如果能通过调整其中一条线的路径避免碰撞，就必须调整——可避免的碰撞是不允许的。**从同一节点出发的多条线**：不要都用 `yshift` 微调同一个锚点——改用不同锚点（如一条从 `.east`，另一条从 `.south east`），让出发方向本身就不同
- [ ] **标注被线穿过检查**：节点下方的补充说明标注是否被向下的连线穿过？如果有，必须把标注合并到框内（`\\ \scriptsize 说明文字`）

### 渲染后视觉对比度（关键教训）

代码中看起来合理的参数，渲染成 300dpi PNG 后经常不可见。以下是经过实际渲染验证的最低参数：

| 元素 | 最低可见参数 | 常见错误 |
|------|-----------|---------|
| 虚线框 | `line width=1.2pt` + 颜色 `!90` | `!60` 或 `!70` 在浅色背景上消失 |
| 激活条/窄矩形 | 宽度 ≥ 0.45cm | 0.3cm 在 PNG 中几乎不可见 |
| 生命线 | `color=xxx!80` | `!60` 在浅色阶段背景上消失 |
| 连线 | `thick`（0.8pt）+ 颜色对比度足够 | 浅橙/浅黄线在白色背景上消失 |
| 标签字号 | `\footnotesize` 以上 | `\scriptsize` 的数学公式在 PNG 中不可读 |
| 标记文字 | `\small\bfseries` 以上（combo标签等） | `\footnotesize` 的 "par"/"loop" 太小 |

**原则**：宁可在代码中感觉"太粗/太深"，也不要在渲染后发现"看不到"。渲染后的视觉效果永远优先于代码中的美感。

### 连线标签与文字防冲突（反复踩坑）

**连线标签（tag）不能骑在其他节点的文字上方**——这是最常见的视觉问题：
- 跨栏/跨层连线的 `tag` 标签放在连线的**空白区域段**上，不放在经过节点附近的段上
- 如果连线经过密集区域（多个节点之间），标签放在连线的起点或终点附近（远离密集区）
- **渲染后逐条检查**：每个 tag 标签是否与其他元素（节点文字、其他标签、连线）重叠
- **有空白空间就不该重叠**——这是原则。只有在确实无空间时才考虑重叠。标签位置选择优先级：
  1. 连线水平段的上方（`above=3pt`）——最自然
  2. 连线竖直段的远离其他元素的一侧（`left=5pt` 或 `right=5pt`）
  3. 弯折点附近的空白区域
  4. 只有以上位置都被占用时才考虑骑在线上（用半透明背景）
- **标签放置后必须检查是否与附近的节点文字重叠**——不要只看标签和连线的关系，还要看标签和周围所有元素的关系。在空间充足时，标签应该放在最远离其他元素的空白处
- **当 `pos` 定位无法找到好位置时，改用绝对坐标的独立 `\node`**——把连线的 `node[tag, pos=...]` 拆成单独的 `\node[tag] at (x, y) {文字};`，这样可以精确放在任何空白处，不受连线路径约束。先看图中哪里有空白，再把标签放到那个坐标
- 如果无法避免冲突，缩短标签文字或移动标签到连线的另一侧（`swap` 选项）
- **标签背景用透明而非白色**：`tag` style 的 `fill=white` 会产生一个白色方块遮挡箭头线，看起来不自然。更好的做法：
  - 首选：标签放在连线**旁边**（用 `above=2pt` 或 `below=2pt`），标签和连线不重叠，无需背景色
  - 次选：如果标签必须在连线上方，用 `fill=white, fill opacity=0.85`（半透明白底），比纯白色自然
  - **禁止**：标签的白色背景把箭头线截断成两段的效果
- **不必要的弯折检查**：当源节点和目标节点的 x 坐标接近（差距 < 1.5cm）时，应该用**直连**（`.south` → `.north`），不要用 L 型绕弯。这种情况用绕弯会产生不自然的多余折线段。渲染后检查：每条连线的弯折次数是否是最少的？有没有可以省掉的弯折？
- **路径注释标签（如"攻击路径""反馈路径"）位置选择**：这类描述整条路径含义的标签不要紧贴弯折点——弯折点附近空间局促容易与其他元素冲突。应放在路径中**最长的直线段**的中点旁边，那里空间最充裕、视觉最自然
