# 画图经验沉淀

> 本文件由 Agent 在画图过程中自动维护。每次遇到编译错误、布局缺陷、或发现有效技巧时，按格式追加记录。
> 下次画同类型图时，先读取本文件获取先验知识。

## 使用规则

### 何时读取
确定图表类型后，先读取本文件中该类型的已有经验。经验标注了发现日期，当作「可能有效的提示」而非「保证正确的事实」——工具版本会更新，之前的 workaround 可能已不需要。

### 何时写入
以下情况自动追加经验记录：
- 编译错误经过 2 次以上尝试才解决
- 发现了某种图表类型的有效布局技巧
- 渲染结果与预期差异大，需要调整方案
- 用户反馈指出了某个反复出现的问题

### 记录格式

每条经验包含：

    ### [图表类型] - [简述]
    - **问题/发现**：具体描述
    - **解决方案**：怎么做的
    - **发现日期**：YYYY-MM-DD

只写经过验证的事实，不写未确认的猜测。如果按经验操作失败，更新或删除该条经验。

## 经验记录

### [通用] - pgfonlayer 环境名
- **问题/发现**：背景层环境名是 `\begin{pgfonlayer}{background}`，不是 `\begin{pgfonbackgroundlayer}`。后者会导致编译报错但仍生成异常 PDF。
- **解决方案**：始终使用 `\pgfdeclarelayer{background}` + `\pgfsetlayers{background,main}` + `\begin{pgfonlayer}{background}`
- **发现日期**：2026-03-28

### [三栏映射图] - zone style 不能在 pgfonlayer 中使用
- **问题/发现**：在 `\begin{pgfonlayer}{background}` 中用 `\node[zone, fit=...]` 时，如果 `zone` style 包含 `inner sep` 等参数，可能导致 zone 框不显示。
- **解决方案**：在 `pgfonlayer` 中直接内联 style（`\fill[dashed, thick, rounded corners=8pt, inner sep=15pt, ...]`），不依赖预定义的 `zone` style。
- **发现日期**：2026-03-28

### [通用] - amsmath 包缺失导致 \boldsymbol 未定义
- **问题/发现**：对比方案图中使用 `\boldsymbol` 但未加载 `amsmath` 包，导致编译报错。
- **解决方案**：在 preamble 中加 `\usepackage{amsmath, amssymb}`，尤其是含数学公式的图。
- **发现日期**：2026-03-29

### [多实例汇聚图] - rotate=90 中文侧栏标签
- **问题/发现**：尝试用 `rotate=90` 放置中文侧栏标签，触发了 xelatex 不可读色块问题。
- **解决方案**：所有中文标注必须水平放置，侧栏标签用多行 `\shortstack` 替代旋转。
- **发现日期**：2026-03-29

### [通用] - stmaryrd 包提供双方括号符号
- **问题/发现**：`\llbracket` / `\rrbracket`（语义双方括号⟦⟧）需要 `stmaryrd` 包，否则编译报错 "Undefined control sequence"。
- **解决方案**：在 preamble 加 `\usepackage{stmaryrd}`。常见于密码学/形式化验证领域的图。
- **发现日期**：2026-03-29

### [时序图] - 生命线必须最后绘制避免被遮挡
- **问题/发现**：生命线虚线被阶段背景 fill、combo 框、注释框遮挡——TikZ 按代码顺序绘制，后画的在上层。
- **解决方案**：将所有 `\draw[lifeline=...]` 移到 `\end{tikzpicture}` 前最后位置。
- **发现日期**：2026-03-29

### [时序图] - 消息标签默认不加白色背景
- **问题/发现**：消息标签 `tag` style 用 `fill=white` 在生命线上产生突兀的白色方块，视觉不自然。
- **解决方案**：`tag` style 默认无背景（透明）。只有标签确实与生命线重叠且影响可读性时，才加 `fill=white, fill opacity=0.85`（半透明）。大多数消息标签放在箭头上方，不需要白底。
- **发现日期**：2026-03-29

### [分层架构图] - x 坐标接近的节点间不要绕弯
- **问题/发现**：两个上下相邻且 x 坐标接近（差距 < 1.5cm）的节点之间，连线用了 L 型绕弯，产生不必要的折线段。
- **解决方案**：x 接近时直连 `.south` → `.north`，不要绕弯。渲染后检查每条连线的弯折次数是否最少。
- **发现日期**：2026-03-29

### [draw.io] - 并列项目不加编号圆
- **问题/发现**：draw.io 路线图中给并列的研究子课题加了 ①②③④ 编号圆，暗示了不存在的顺序关系，看起来突兀。
- **解决方案**：默认不加编号。只有当项目之间有明确顺序/优先级关系时才加。
- **发现日期**：2026-03-29

### [通用] - 非90度弯折（路径中间点坐标未对齐）
- **问题/发现**：连线弯折处不是标准90度角，出现奇怪的倾斜弯折。原因是路径中间点的 x 或 y 没有与前/后路径点精确对齐。
- **解决方案**：代码写完后逐条 `\draw` 检查：路径中相邻两个坐标点必须共享 x 或 y。不共享的就是隐式斜线。例如从 `(2,10)` 到 `(7,8)` 中间点必须是 `(2,8)` 或 `(7,10)`。
- **发现日期**：2026-03-29

### [几何图] - 坐标标签与曲线微重叠
- **问题/发现**：椭圆曲线图中点坐标标注 `(1,-1)` 和点名称 `R'` 与曲线有轻微重叠，图例框和步骤框也有微重叠。审查 agent 未发现。
- **解决方案**：标注放在曲线外侧（凹面方向），多个信息框之间间距 ≥ 0.3cm。审查时对"微重叠"（1-2px 级别）保持敏感。
- **发现日期**：2026-03-29

### [draw.io] - 箭头线覆盖文字（z-order 问题）
- **问题/发现**：draw.io 导出的图中蓝色箭头线显示在模块文字上方，文字被箭头遮挡。
- **解决方案**：在 XML 中先定义 edge（连线），后定义 vertex（框和文字），确保框和文字在更高图层。
- **发现日期**：2026-03-29

### [多实例汇聚图] - 迭代修改破坏了平行关系
- **问题/发现**：迭代修改中红色虚线和蓝色实线原本是平行的（美观），修改后不再平行（退步）。审查 agent 未发现这种退步。
- **解决方案**：每次迭代修改后，检查修改是否破坏了已有的好效果（对称性、平行、间距均匀度）。不允许修复一个问题引入另一个审美问题。
- **发现日期**：2026-03-29

### [通用] - 箭头出发/到达锚点导致"看起来指反了"
- **问题/发现**：分层架构图、数据流水线图、三栏映射图中，多条箭头的视觉流向与逻辑数据流方向矛盾。原因是出发锚点和到达锚点选错了——比如目标在上方却从 `.east` 出发，导致第一段线段水平走，读者觉得箭头"先往右走了"而不是"向上流"。这是 AI 最常忽略但人一眼就能看出的问题。
- **解决方案**：出发锚点的方向必须朝着目标所在方向（目标在上方就从 `.north`，在右边就从 `.east`）。多条线避免重叠时，用同方向的相邻锚点（`xshift`/`yshift` 微调），而不是换到反方向的锚点。代码写完后逐条 `\draw` 检查第一段线段方向是否和逻辑方向一致。
- **发现日期**：2026-03-29

### [时序图] - 消息标签颜色不要五颜六色
- **问题/发现**：时序图中每个参与方的消息标签用了不同颜色（绿色、橙色、紫色等），整体看起来像圣诞树，不专业。
- **解决方案**：消息标签文字统一用 `black!80`，只有关键/异常消息用红色。箭头线颜色可以区分参与方，但标签文字颜色保持统一。
- **发现日期**：2026-03-29

### [几何图] - 辅助标注箭头不要死板指向正中心
- **问题/发现**：Merkle 树图中，"兄弟节点"标签的箭头死板地指向 H0 的正底部（.south），从左下方斜着插上去看起来很僵硬。
- **解决方案**：辅助标注箭头可以指向元素偏左/偏右的位置（如 `.south west` 或 `[xshift=-3pt]node.south`），不需要精确对准正中心。像人手画的标注一样自然，不要像机器一样死板。
- **发现日期**：2026-03-29

### [通用] - 同类线条从同一点出发更优雅
- **问题/发现**：对比图和映射图中，两条同类型实线被强制从不同锚点出发（遵循"用不同锚点"规则），但结果反而让路径变得扭曲不自然。
- **解决方案**：同类型线（同色、同线型、去往相似方向）可以共享出发点——看起来更像"有意分发"。只有不同类型线（实线 vs 虚线）才需要分开。规则不是死的，美感才是最终裁判。
- **发现日期**：2026-03-29

### [分层架构图] - 反馈虚线穿过大量文字
- **问题/发现**：应急响应编排到安全策略编排的反馈虚线穿过了多个层的文字标签，审查 agent 未发现。
- **解决方案**：虚线/反馈线是最容易出问题的线——必须逐段检查路径上有没有文字。虚线穿过文字是最明显的排版问题，哪怕只穿过一个标签也必须修改路径绕开。
- **发现日期**：2026-03-29

### [几何图] - 大括号方向搞反
- **问题/发现**：注意力机制图中 Q/K^T/V 的大括号开口方向不对，看起来很怪。
- **解决方案**：大括号 `{` 开口朝右标注右侧内容，`}` 开口朝左。如果方向不确定或效果不好，直接去掉改用文字标注（如在框旁写 $d_k$）——少一个有问题的装饰比多一个更好。
- **发现日期**：2026-03-29

### [draw.io] - 背景色跑到箭头上方
- **问题/发现**：draw.io 路线图中区域背景色覆盖在箭头上方，箭头被"吃掉"了。
- **解决方案**：XML 中 z-order 三层结构：背景（最先定义）< 连线 < 框体/文字（最后定义）。背景 mxCell 必须在 edge 之前定义。
- **发现日期**：2026-03-29

### [通用] - "能直就直"——对齐节点间不要加不必要弯折
- **问题/发现**：对比图和 draw.io 路线图中，源和目标完全垂直对齐，但箭头还是画了 L 型弯折。人一看就觉得"这个弯折是多余的"。
- **解决方案**：画连线前先检查源和目标的 x/y 是否对齐。对齐则直线，不对齐才弯折。每次弯折必须有理由（绕障碍物），没理由的弯折就是多余的。
- **发现日期**：2026-03-29

### [通用] - 弯折后线段太短导致箭头断在弯角
- **问题/发现**：对比图中从全局模型到评估指标的连线，弯折点紧挨着箭头尖，看起来像箭头断在了弯角处。
- **解决方案**：弯折点到箭头尖之间的线段长度 ≥ 0.8cm。如果空间不够，提前弯折让最后一段足够长。
- **发现日期**：2026-03-29

### [通用] - 标签有空间却还是重叠放置
- **问题/发现**：量化流程图中 Yes/No 标签和线重叠，hub-spoke 图中标签和框重叠——明明旁边有空白空间。
- **解决方案**：放置每个标签时必须看周围环境——有空白就挪到空白处。"有空间却重叠"是最不应该的错误，说明画图时根本没看周围。
- **发现日期**：2026-03-29

### [通用] - 箭头指向空气（没"够到"目标）
- **问题/发现**：时序图中自调用弧太短，终点在激活条外面——箭头"够不到"目标元素。
- **解决方案**：确保箭头终点落在目标元素的边框上。如果目标太小导致够不到，扩大目标元素（加宽激活条、加长框体），不要缩短箭头。

### [draw.io] - 内容在容器框中不居中
- **问题/发现**：draw.io 路线图中内容偏左或偏上，在容器框内不居中。
- **解决方案**：容器框内的内容必须水平和垂直居中。计算容器中心和内容中心的偏差，调整到一致。
- **发现日期**：2026-03-29

### [分层架构图] - 图片被截断（border 不够）
- **问题/发现**：架构图右侧被截断——border=35pt 仍然不够，因为跨层 rail 延伸到了最右侧。
- **解决方案**：border 必须覆盖所有元素。画完后检查最右/最左/最上/最下的元素坐标，确保 border 足够。如果有 rail 或反馈线在画面边缘，border 至少要 `max(元素最远距离) + 25pt`。
- **发现日期**：2026-03-29

### [通用] - 箭头中间路径点超越目标导致方向反转
- **问题/发现**：Transformer 架构图中 `\draw[arrow] (pe.north) -- (-5.5, 5.65) -- (mha.south)` ——中间点 y=5.65 超过了目标 mha.south 的 y=5.4，最后一段路径从 5.65→5.4 向下，箭头尖朝下，但数据应向上流。箭头连对了地方但头朝反了——最阴险的方向 bug。
- **解决方案**：同轴路径（x 相同）删掉多余中间点直连。不同轴时确保每个中间点坐标值严格在前一点和目标之间（单调递增/递减）。
- **发现日期**：2026-03-31

### [通用] - 箭头头大小与箭身长度不匹配
- **问题/发现**：zkSNARK 图中 Prover→Verifier 的短箭头（0.5cm）用了 `Stealth[scale=1.5]`，渲染后是一个实心紫色三角形而非箭头。同一张图中三个功能相似的箭头（Commit、Prove、Prover→Verifier）分别用了 scale=1.2/1.5/0.8，粗细不一致。
- **解决方案**：(1) 箭身 ≥2cm→scale≤1.2；1-2cm→scale≤0.9；0.5-1cm→scale≤0.7。(2) 同类功能箭头必须统一 scale 和 line width——全局 style 中的 scale 值要适配最短的那条箭头。
- **发现日期**：2026-03-31

### [通用] - 区域边框 padding 过大产生空带
- **问题/发现**：Transformer 图 Decoder zone 在 Layer 6 上方留了 2.2cm 空带，zkSNARK 图 phase 背景下方延伸到 y=-12.5 但内容只到 y=-9.0。大面积区域内空白让图显得松散。
- **解决方案**：区域边框（zone/region/phase background）到最外层内容的 padding ≤ 1cm。画完后检查每个区域：边框内最远的内容到边框的距离是否合理？
- **发现日期**：2026-03-31

### [通用] - 小标注过多导致全图重叠（设计野心过度）
- **问题/发现**：区块链架构图中，模型为追求信息密度，在每个框旁边加了协议标注（.sol、sDfx invoke、inter blockchain、pin-pinning、REST/gRPC 等），但这些标注挤在框边和 zone 边框上，导致 7 处以上重叠。标注越多重叠越多。

### [通用] - 树状分叉处线段断连和空心方块
- **问题/发现**：树状分叉（一分多）的主干、横杆、分支用多个独立 `\draw` 画，每个都继承了全局 `shorten >=2pt`，导致在交叉点处每段线各自缩短 2pt，出现可见间隙。更严重的是有时分叉点用了小 circle/rectangle 节点，渲染为"空心方块"。
- **解决方案**：(1) 主干+横杆用一条连续 `\draw[thick]` 路径画完，**不带 arrow、不带 shorten**；(2) 只有最终指向目标的分支才用 `\draw[arrow]`（继承 shorten）；(3) 分叉点用 `coordinate` 而非 `\node`——coordinate 不渲染任何形状。
- **发现日期**：2026-04-03
